A Perceptually Uniform Color System for Theme 0: Strategy, Palettes, and Token Architecture
Section 1: Foundations of a Modern, Perceptually-Uniform Color System
This section establishes the strategic rationale for the proposed color system. It details the technical superiority of the OKLCH color model, defines the accessibility standards that govern the project, and addresses the practical realities of the current design and development ecosystem. This foundation provides the necessary context for the concrete palette and token recommendations that follow.

1.1 The Strategic Imperative for OKLCH
The selection of a color model is a foundational architectural decision for any design system. Adopting the OKLCH color space represents a strategic upgrade over legacy models like RGB and HSL. This choice is not merely stylistic; it aligns the technical representation of color with the mechanics of human perception, forming the cornerstone of an accessible, intuitive, and maintainable design system.   

Perceptual Uniformity and Predictability
The principal advantage of OKLCH is its perceptual uniformity. In this model, discrete mathematical adjustments to the Lightness (L), Chroma (C), or Hue (H) values produce visually consistent and predictable changes. For example, increasing the Lightness value by 0.1 will result in a perceptually equivalent lightening of a color, regardless of its hue. This stands in stark contrast to the HSL model, where the lightness parameter is not tied to human perception. In HSL, a 10% change in lightness for a yellow color can appear dramatically different from a 10% change for a blue color, leading to unpredictable and often disharmonious results when generating color scales or interactive states.   

This predictability is paramount for a design system, as it allows for the programmatic generation of entire color palettes and UI states. A designer or developer can confidently create a darker or lighter variant of a base color simply by adjusting the L value, knowing the result will be a natural, consistent variation without unexpected shifts in saturation or hue. This directly addresses the requirement for "predictable deltas" across all contexts.   

Expanded Color Gamut (Display-P3)
Legacy color models such as RGB and HSL are fundamentally tied to the sRGB color gamut. The OKLCH model is gamut-agnostic, meaning it can define colors that exist outside of sRGB. This provides access to the wider Display P3 gamut, which is supported by most modern Apple devices and other high-end displays, and contains approximately 50% more colors than sRGB. This capability future-proofs the design system, enabling the use of richer, more vibrant accent colors that can enhance visual appeal and brand expression on capable hardware, while still allowing for graceful degradation to the closest sRGB equivalent on older screens.   

Enhanced Developer Experience and Maintainability
The OKLCH syntax, oklch(L C H), is inherently more human-readable than hexadecimal or RGB values. A developer can inspect a value like oklch(0.8 0.02 300) and immediately understand that it is a very light, low-saturation color in the purple-pink range. This clarity simplifies the process of debugging accessibility issues, as a low contrast between a background of oklch(0.8...) and a text color of oklch(1...) is immediately apparent from the code itself.   

Furthermore, modern CSS specifications provide a relative color syntax, oklch(from var(--base-color) calc(l + 0.1) c h), which allows for the dynamic modification of colors directly in the browser. This is a transformative feature for design systems. Instead of defining separate color variables for every interactive state (e.g., --button-background-hover, --button-background-active), a single rule can programmatically generate these states from a base token. This drastically reduces the number of color tokens that need to be defined and maintained, leading to a leaner, more scalable, and more robust CSS architecture.   

1.2 Defining the Accessibility Mandate: WCAG 2.2 AA and Beyond
A modern color system must be accessible by design, not by exception. The non-negotiable standard for this project is the Web Content Accessibility Guidelines (WCAG) 2.2, at the conformance level of AA. This standard provides clear, testable criteria for ensuring that visual information is perceivable by users with various forms of visual impairment.

The WCAG 2.2 AA Compliance Baseline
The system must satisfy the following key success criteria related to color contrast:

Text Contrast (Success Criterion 1.4.3): The contrast ratio between text and its background must be at least 4.5:1 for normal-sized text. For "large text," defined as 24px and larger, or 18.66px and larger if bold, the minimum required contrast ratio is 3:1.   

Non-Text Contrast (Success Criterion 1.4.11): The contrast ratio for user interface components (such as input borders and icons) and meaningful graphical objects must be at least 3:1 against adjacent colors. This ensures that interactive elements and important information are clearly distinguishable.   

Focus Appearance (Success Criterion 2.4.13): The focus indicator for interactive elements must be clearly visible. While this criterion has multiple facets, it includes a contrast requirement that the indicator has a contrast ratio of at least 3:1 against the colors adjacent to it.   

These ratios are absolute thresholds; a value of 4.49:1 constitutes a failure. All palettes and token assignments proposed in this report will be rigorously validated against these criteria.   

Augmenting Compliance with APCA for Perceptual Robustness
While WCAG 2.2 AA provides the necessary baseline for legal and standards compliance, its underlying contrast calculation algorithm has well-documented flaws. The algorithm can perform poorly, particularly for colors on dark backgrounds, and has been shown to pass color combinations that are perceptually difficult to read while failing others that are perfectly legible. This is because the WCAG 2.x formula is a relatively simple mathematical ratio of luminance that does not fully account for the complexities of human visual perception.   

To address these shortcomings and build a system that is not just compliant but truly accessible, this strategy will employ the Accessible Perceptual Contrast Algorithm (APCA) as a secondary quality heuristic. APCA is an emerging, next-generation contrast model designed to replace the WCAG 2.x algorithm in the future (likely in WCAG 3.0). It provides a much more accurate prediction of perceived text legibility across different contexts, including font weight, size, and color polarity (light-on-dark vs. dark-on-light).   

The strategic approach is therefore twofold:

Achieve provable WCAG 2.2 AA compliance across the entire system to meet all current standards and legal requirements.

Use APCA as a "sense check" to validate key color pairings, especially for body text and in dark mode contexts. This dual-pronged validation will identify and eliminate any "false passes" allowed by the older WCAG algorithm, ensuring the final color system delivers a high-quality, perceptually robust experience for all users.

1.3 Navigating the Ecosystem: Browser Support and the Design Tool Gap
A color strategy is only viable if it can be implemented effectively with available tools. The current ecosystem presents a mixed landscape for OKLCH adoption, with strong support in browsers but a significant gap in mainstream design software.

Browser Readiness for Production
The oklch() color function is fully supported in all modern evergreen browsers, including Chrome, Firefox, Safari, and Edge. Global browser support now exceeds 92%, making it a production-ready feature for web development. The powerful relative color syntax (from), which is critical for efficient state management, also enjoys strong support at approximately 88% and is on a clear path to ubiquity. This level of support provides high confidence for building a web-based design system founded on OKLCH.   

The Design Tool Bottleneck and a Proposed Workflow
The most significant challenge in the current OKLCH workflow is the lack of native support in dominant UI design tools, most notably Figma. As of now, Figma's color pickers, styles, and variables do not natively support the OKLCH color space. This creates a critical disconnect between design and development. Designers working in HSL or RGB are not operating in the same perceptually uniform space as the final implementation, which can lead to "translation nightmares," where colors specified in a design file render with unexpected perceptual lightness or saturation once converted to OKLCH in code.   

To bridge this gap and maintain the integrity of the color system, a structured, developer-driven workflow with a clear designer contract is required. This process ensures that the OKLCH color model remains the single source of truth throughout the entire product development lifecycle.

The recommended workflow is as follows:

Centralized Palette Generation: The master color palette is defined not within Figma, but using external, OKLCH-aware tools. Tools like Atmos, Harmonizer, or the OKLCH Palette Generator are designed specifically for this purpose, allowing for the creation of perceptually uniform ramps. The output of this stage—a definitive list of OKLCH color values—becomes the system's immutable source of truth.   

Designer Contract and Tooling: Designers operate under a clear contract: they are to use colors from the pre-defined master palette, not create new ones ad-hoc within Figma. To facilitate this, they will use specialized Figma plugins like OkColor or Polychrom. These plugins provide an OKLCH-native color picker within Figma, allowing designers to accurately apply the exact values from the master palette to their designs.   

Developer Implementation: Developers consume the master palette directly, typically as a set of design tokens (e.g., in a JSON or CSS file). They can implement these values with high confidence, knowing that the OKLCH values in the Figma mockups—applied via the plugin—are an accurate representation of the final implementation.

This structured process mitigates the current limitations of design tooling, preventing color drift and ensuring that the perceptual benefits of OKLCH are preserved from initial concept to final product.

Section 2: Theme 0 Palette Candidates & Accessibility Analysis
This section presents two distinct, fully realized color palette candidates for Theme 0. It details the systematic methodology used to generate them and provides a rigorous, data-driven analysis of their compliance with WCAG 2.2 AA accessibility standards.

2.1 Methodology for Palette Generation
The palettes were generated using a systematic, repeatable process designed to leverage the perceptual uniformity of the OKLCH color space. This approach moves away from manual color picking and toward a more engineered system that guarantees consistency and harmony.

The methodology consists of three primary steps:

Anchor Selection: For each palette, a set of foundational "anchor" colors was chosen. This includes a primary brand hue, a neutral hue for backgrounds and text, and the four standard semantic hues for system status: blue for information, green for success, yellow/orange for warning, and red for critical.

Perceptually Uniform Ramp Generation: For each anchor hue, a 10-step color ramp was generated using an OKLCH-native palette tool. This process involves three key constraints to ensure perceptual uniformity:   

The Hue (H) is held constant across all 10 steps of a single ramp.

The Chroma (C) is kept as consistent as possible across the ramp to maintain a similar level of saturation and vibrancy. Minor adjustments are made to stay within the sRGB gamut at very light or dark extremes.

The Lightness (L) is systematically varied in perceptually even increments, typically ranging from a very light value (e.g., L=0.98) to a very dark value (e.g., L=0.15). This is the primary driver of the ramp's steps and ensures that the visual distance between Color-100 and Color-200 is perceived as being the same as the distance between Color-800 and Color-900.   

Neutral Axis Creation: A separate neutral (gray) ramp was generated by setting the Chroma (C) to a very low value (near zero). This creates a set of grays that can be tinted slightly by the primary brand hue for greater harmony, while serving as the primary basis for surfaces and text.

Section 3: A Scalable Architecture for Semantic Color Tokens
A robust color system is defined not by its palette alone, but by the intelligence of its architecture. This section outlines a scalable, semantic token architecture that decouples the intended purpose of a color from its literal value. This abstraction is the key to creating a system that is maintainable, themeable, and easy for both designers and developers to use consistently.

3.1 The Principle of Semantic Abstraction
The architecture will employ a multi-tiered token system, with a primary focus on semantic tokens. This approach contrasts with using primitive or direct value tokens (e.g., blue-500) in application code.   

Primitive Tokens: These are the direct, value-based representations of the palette (e.g., --color-neutral-100, --color-primary-500). They represent the "what."

Semantic Tokens: These are role-based, abstract names that describe a color's purpose in the UI (e.g., --color-background-page, --color-text-interactive). They represent the "why".   

The core principle is that components and application code should only reference semantic tokens. The semantic tokens, in turn, reference the primitive tokens. This layer of abstraction provides two critical advantages:

Maintainability: If the brand's primary blue changes, only the mapping of a few semantic tokens (like --color-background-interactive) needs to be updated in a single location. Without this, every instance of blue-500 would need to be found and replaced across the entire codebase.   

Themeability: Creating a dark mode or any other theme becomes a matter of providing a different set of mappings for the same semantic tokens. The component code remains unchanged because it only knows about the role (e.g., background-color: var(--color-background-page)), not the specific value.   

The naming convention will follow a structured [category]-[property]-[variant]-[state] pattern for clarity and predictability.

3.2 Surface Tokens: The Canvas of the UI
Surface tokens define the background colors of the user interface, establishing its fundamental structure and visual hierarchy. They form the canvas upon which all other elements are placed.

surface-page: The base background color for an entire page or view. This is the most foundational layer.

surface-primary: The primary surface color for main content containers like cards, modals, and sidebars. It typically sits atop the surface-page.

surface-secondary: A surface color used for secondary content areas or to differentiate sections within a primary container.

surface-interactive-default: The default background color for primary interactive components, such as buttons and toggles.

surface-interactive-hover: The background color for a primary interactive component in its hover state.

surface-interactive-pressed: The background color for a primary interactive component in its pressed or active state.

surface-disabled: The background color for disabled components, indicating non-interactivity.

3.3 Text & Icon Tokens: The Content Layer
These tokens define the colors for all foreground content, including typography and iconography. Their names are designed to communicate their intended use and their relationship with the surface tokens, ensuring accessible pairings.

text-primary: The color for the most prominent text, such as page titles and headings.

text-secondary: The color for standard body copy and less-emphasized text content.

text-subtle: The color for tertiary text, such as helper text, metadata, or placeholder content in forms.

text-on-interactive: The color for text that is placed on top of an interactive surface (e.g., the label on a primary button). This token is critical for ensuring contrast against colored backgrounds.

text-disabled: The color for text within disabled components.

icon-primary: The color for primary, standalone icons.

icon-secondary: The color for less prominent icons or those paired with secondary text.

icon-on-interactive: The color for icons placed on an interactive surface.

3.4 Status Tokens: Communicating System State
A comprehensive and consistent set of status tokens is essential for communicating system feedback to the user in a predictable and accessible way. This system provides a full suite of role-based tokens for each of the four standard statuses, inspired by the robust systems of Atlassian. This prevents designers and developers from creating ad-hoc, inconsistent, and potentially inaccessible status indicators.   

status-info-surface: The background color for informational banners, alerts, or components.

status-info-text: The text color for use on an info surface.

status-info-icon: The icon color for use in info contexts.

status-info-border: The border color for info components.

This four-part structure is repeated for the other statuses:

status-success-{surface, text, icon, border}

status-warning-{surface, text, icon, border}

status-critical-{surface, text, icon, border}

This consistent structure ensures that a "success" state is always rendered with the same accessible combination of a light green background, dark green text, and a corresponding icon color, enforcing UI consistency and compliance across the entire application.

3.5 Focus Tokens: Ensuring Navigational Clarity
Clear and visible focus indicators are a cornerstone of keyboard accessibility. This system will implement the W3C's recommended technique for creating a universally high-contrast focus indicator by using a two-color ring. This approach solves the common problem where a single-color focus ring fails contrast requirements when an element is placed on a background of a similar color.   

The technique relies on two colors that have a very high contrast ratio with each other (e.g., >9:1). This guarantees that at least one of the two colors will always have a sufficient contrast ratio (≥3:1) against any solid background color the element might be placed on.

focus-ring-outer: The outer color of the two-color focus ring. This is typically a dark, high-contrast color.

focus-ring-inner: The inner color of the two-color focus ring. This is typically a very light, high-contrast color (like white).

focus-ring-inset: A CSS value for the offset of the focus ring, allowing it to appear inside or outside the component's boundary.

This robust, technical solution ensures that focus is never lost, providing a highly accessible navigation experience for all users, regardless of the complexity of the UI's background colors.

4.3 Implementation Guidance
The token system is designed for direct implementation using modern CSS features.

CSS Custom Properties
The master assignment table should be translated directly into a :root declaration block in a global CSS file. This makes the entire color system available throughout the application.

CSS

:root {
  /* Surface Tokens */
  --surface-page: oklch(0.99 0.002 250);
  --surface-primary: oklch(0.98 0.005 250);
  --surface-interactive-default: oklch(0.60 0.15 250);

  /* Text & Icon Tokens */
  --text-primary: oklch(0.22 0.015 250);
  --text-on-interactive: oklch(0.98 0.005 250);

  /*... all other tokens from the master table */
}
Programmatic States with Relative Color Syntax
Leverage the oklch(from...) syntax to generate interactive states programmatically, reducing token redundancy. The hover and pressed states for interactive surfaces, for instance, should not be defined as separate primitive colors but generated from the default state. The values in the master table for hover and pressed states serve as a reference, but the implementation should be dynamic.

CSS

.button-primary {
  background-color: var(--surface-interactive-default);
  color: var(--text-on-interactive);
}

.button-primary:hover {
  /* Programmatically darken the default state by reducing lightness */
  background-color: oklch(from var(--surface-interactive-default) calc(l - 0.06) c h);
}

.button-primary:active {
  /* Further darken for the pressed state */
  background-color: oklch(from var(--surface-interactive-default) calc(l - 0.12) c h);
}
This approach ensures that if --surface-interactive-default is ever updated, its hover and active states will update automatically and maintain the same perceptual distance, fulfilling the "predictable deltas" requirement.   

Theming Strategy for Dark Mode
The semantic token architecture is the key to efficient theming. A dark mode can be implemented simply by re-mapping the semantic tokens to different primitive values from the palette within a prefers-color-scheme: dark media query. The application's component styles do not need to change.

CSS

@media (prefers-color-scheme: dark) {
  :root {
    /* Remap surface tokens for dark mode */
    --surface-page: oklch(0.18 0.015 250); /* Was Neutral-50, now Neutral-950 */
    --surface-primary: oklch(0.22 0.015 250); /* Was Neutral-100, now Neutral-900 */
    
    /* Remap text tokens, inverting lightness */
    --text-primary: oklch(0.98 0.005 250); /* Was Neutral-900, now Neutral-100 */
    --text-secondary: oklch(0.75 0.015 250); /* Was Neutral-700, now Neutral-400 */

    /* Remap interactive tokens for a dark context */
    --surface-interactive-default: oklch(0.60 0.15 250); /* Stays Primary-500 */
    --text-on-interactive: oklch(0.98 0.005 250); /* Still light text for contrast */

    /*... remap all other tokens as needed for dark theme */
  }
}
This demonstrates the power of the abstraction layer, enabling complex themes to be managed with a small, centralized block of code.   

4.4 Concluding Remarks and Next Steps
The proposed color system provides a robust, scalable, and accessible foundation for Theme 0. By building upon the perceptually uniform OKLCH color space and a semantic token architecture, it achieves accessibility by design, simplifies maintenance, and enables powerful features like programmatic state generation and streamlined theming. It represents a modern, engineering-led approach to color that aligns technical implementation with human perception.

Next Steps:

Dissemination: The master token assignment table should be disseminated to all design and engineering teams as the single source of truth for color.

Tooling Integration: The recommended Figma workflow, utilizing OKLCH-aware plugins, should be adopted by the design team to ensure fidelity between design mockups and production code.

Component Implementation: Begin the process of migrating existing UI components or building new ones to consume the new semantic color tokens.

Documentation: The token names, their intended usage, and the rationale behind the system should be formally documented as part of the official design system guidelines.
