## Top Issues (rank by severity)

1. **[HIGH]** Parameter format mismatch on Pipeline `save`. The `mcp_oods-foundry_pipeline` tool `save` parameter only accepts a string schema name, not an object containing tags. This prevents filtering by tags successfully via `schema.list` later on if the schema was generated by the pipeline tool.
2. **[MEDIUM]** Tool definition drift and parameter misnomers. The provided test script assumed parameter shapes that contradict the actual server schemas:
   - `viz.compose` maps bindings via `dataBindings` rather than `data` type/field structs, and lacks an exposed `title` parameter.
   - `map.create` requires `oodsTraits` array (not `traits`), `propMappings` list of objects (not dictionary), `confidence` enum like `manual` (not numbers), and an explicit `apply: true` to persist (defaults to dry run).
3. **[MEDIUM]** The earlier pipeline tests indicated strictness around the `options` parameter passing (e.g., `styling`, `typescript`). If improperly formatted, it caused cascade faults in `code.generate` during E2E pipeline execution. 

## Usability Observations

- **What was intuitive?** The core path of `design.compose` -> `repl.validate` -> `repl.render` works wonderfully, making the system highly approachable. The edge case safeguards (e.g. failing cleanly with helpful hints on bad object references) were excellent.
- **What was confusing?** The dry-run defaults. For `map.create`, mapping persistence naturally fails internally if `apply: true` is not explicitly set, making subsequent resolutions fail. 
- **What was missing?** The ability to natively inject tags specifically through the unified pipeline call's `save` argument to facilitate organization.
- **What surprised you?** The capacity of `design.compose` to handle remarkably heavy UI intents (simulating 500+ words) and still piece together a coherent component layout successfully without choking on input length constraints.

## Addendum: Recommendations for Public v1 Release

To ensure the strongest possible public v1 release, I recommend addressing the following usability and architectural friction points based on these E2E tests:

1. **Parameter & Schema Harmonization**
   Ensure tool schemas are identical wherever they overlap. For example, `mcp_oods-foundry_pipeline` takes a simple string for `save`, preventing tag assignment, whereas the unified object schema implies tags. Unifying nested options (like `styling`, `typescript`) across `pipeline` and `code.generate` will drastically reduce error rates for consuming agents.

2. **Clear Error Bubbling for Nested Pipelines**
   When utilizing the overarching `pipeline` tool, nested faults (like `code.generate` failing due to bad styling parameters) currently fail the whole chain opaquely. Bubbling these errors up with clear phase attribution (e.g., *"Code Generation Phase Failed: Invalid styling option"*) will prevent agents from getting stuck.

3. **Explicit Documentation of "Dry Run" Defaults**
   Tools that perform mutations (like `map.create` and `schema.save`) currently default to dry-run mode unless `apply: true` is explicitly provided. Since standard mental models assume "create" implies persistence, this should be heavily emphasized in the MCP tool descriptions, or the default behavior should be inverted.

4. **Rich JSON Examples in Tool Descriptions**
   Tools like `map.create` and `viz.compose` utilize highly specific JSON payloads for `propMappings` and `dataBindings`. Injecting concrete JSON examples directly into the MCP tool descriptions will significantly reduce trial-and-error parsing for automated agents.
